Problem

Braud Property Management runs on Buildium's Growth plan — which means no API. Every morning, someone logged in, opened 17–19 separate operational reports, downloaded them, and stitched together answers ownership needed: who's delinquent, what's vacant, which maintenance tickets are overdue. The reports existed. The manual assembly didn't scale.

Approach

I built two connected systems instead of one monolith:

The Braud Pipeline — nightly Playwright automation that logs into Buildium, downloads each report, normalizes the exports in Python, and loads a stable PostgreSQL schema. Prefect handles scheduling (02:00 CST), retries, and failure alerts.

01

Extract

Playwright headless browser — login, navigate, download. No API required.

02

Transform

Python normalizes formats, deduplicates rows, validates before load.

03

Load

PostgreSQL schema for occupancy, delinquency, maintenance, and leases.

Braud Operations Dashboard — a UI that reads that database and gives ownership one place to start the day: Morning Briefing, Executive KPIs, Decision Inbox, and Occupancy modules. The portfolio demo on this site uses fictional data; production reads live pipeline output.

Morning Briefing

What needs attention first — delinquency, vacancies, overdue maintenance, expiring leases.

Executive KPIs

Occupancy, collection rate, delinquent balance, vacant unit count.

Decision Inbox

Drafted actions ownership can copy into Buildium, then mark complete.

Occupancy Snapshot

Portfolio-wide and per-property unit status.

Decision

Playwright over waiting for an API upgrade. Upgrading Buildium plans for API access was a recurring cost with an uncertain timeline. Browser automation is brittle, but it was shippable in weeks and entirely under our control. I built retry logic and report-level error isolation so one failed export doesn't block the rest of the nightly run.

PostgreSQL as the single source of truth. Power BI was in the stack, but ownership needed a purpose-built morning workflow — not a generic report gallery. A custom dashboard on top of Postgres let me design around the four questions they actually ask first, instead of fitting their process into a template.

Result

Both systems are live in production. The pipeline runs nightly with zero manual steps after deployment, covering 17–19 operational reports. Ownership opens the dashboard each morning instead of assembling spreadsheets.

Ownership opens the dashboard each morning with delinquency, vacancy, and maintenance status already assembled — instead of downloading and stitching together 17+ Buildium reports by hand.
